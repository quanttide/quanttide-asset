# 飞书 Wiki 资产验证规则建模

---

## 建模思路

### 核心问题
如何从飞书 Wiki 扫描结果中，提炼出可验证的通用资产模型？

### 建模框架（四步）

```
┌─────────────────────────────────────────────────────────────┐
│                     建模框架                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 业务拆解                                                │
│     └→ 从数据中识别节点类型                                   │
│                                                             │
│  2. 行为定义                                                │
│     └→ 定义每种节点的属性和关系                               │
│                                                             │
│  3. 度量设定                                                │
│     └→ 设定可验证的维度（结构/命名/关系/层级）               │
│                                                             │
│  4. 模型产出                                                │
│     └→ 输出通用模型 + YAML 验证规则                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 跨 Space 的通用性

观察了 4 个不同类型的 Space，发现**结构本质上是一样的**：

| Space | 结构特征 | 分类命名 | 叶子文档命名 |
|-------|----------|----------|--------------|
| 议事档案 | 首页 → 分类 → 子分类 → 叶子 | 议题/议程/决策/计划 | `YYYY年第N周-类型K-主题` |
| 协作档案 | 首页 → 分类 → 列表 → 叶子 | 任务管理/任务列表 | `YYYY年第N周-任务K-主题.docx` |
| 图书馆 | 根分类 → 子分类 → 叶子 | 编程语言/Android/C/Flutter | 书名.pdf |
| 软件库 | 根文件 → 分类 → 叶子 | SSH软件/Python及IDE | 软件名.exe |

**结论：所有 Space 都遵循同一棵树的逻辑，只是叶子节点的内容不同。**

## 1. 业务拆解

### 1.1 节点类型识别

从元数据中识别出 3 种节点类型：

```
┌─────────────────────────────────────────────────────────────┐
│                    节点类型（跨 Space 通用）                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Container（分类节点）                                        │
│  ├─ 特征：has_child = true                                  │
│  ├─ obj_type：docx 或 folder                                │
│  └─ 作用：组织结构，可以有子节点                             │
│                                                             │
│  Leaf（叶子节点）                                            │
│  ├─ 特征：has_child = false                                 │
│  ├─ obj_type：docx 或 file                                  │
│  └─ 作用：存储实际内容                                       │
│                                                             │
│  Root（根节点）                                              │
│  ├─ 特征：parent_token = ""                                 │
│  └─ 作用：Space 的入口点                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 节点元数据

```json
{
  "node_token": "xxx",
  "title": "节点标题",
  "obj_type": "docx | file | folder",
  "parent_token": "父节点token（根节点为空）",
  "has_child": true | false,
  "updated_at": "更新时间"
}
```

## 2. 行为定义

### 2.1 节点属性

| 属性 | 类型 | 说明 |
|------|------|------|
| `title` | string | 节点标题 |
| `obj_type` | enum | docx/file/folder |
| `has_child` | boolean | 是否有子节点 |
| `parent_token` | string | 父节点 token |
| `depth` | int | 深度（根=0） |

### 2.2 节点关系

```
Space
└── Root（depth=0, parent_token=""）
    └── Container（depth=1, has_child=true）
        └── Container 或 Leaf（depth=2+）
            └── Leaf（depth=N, has_child=false）
```

### 2.3 分类模式

| 模式 | Space 示例 | 分类特点 |
|------|-----------|----------|
| 固定分类 | 议事档案 | 分类名固定：议题/议程/决策/计划 |
| 列表分类 | 协作档案 | 分类名动态：任务管理/任务列表 |
| 主题分类 | 图书馆 | 按主题：编程语言/Android/C |
| 功能分类 | 软件库 | 按功能：SSH软件/Python及IDE |

**通用规则：分类节点的 title 不能为空。**

## 3. 度量设定

### 3.1 可验证的维度

| 维度 | 可验证项 | 说明 |
|------|----------|------|
| **结构** | Space 必须有根节点 | depth=0, parent_token="" |
| **结构** | 根节点不能有子节点 | has_child=false |
| **结构** | Container 必须有子节点 | has_child=true |
| **结构** | Container 的 parent 必须存在 | 父子关系完整 |
| **命名** | title 不能为空 | 空标题无法识别 |
| **命名** | title 符合 Space 特有的命名规范 | 可选，建议验证 |
| **关系** | has_child 与实际子节点数量一致 | 标记准确性 |

### 3.2 度量级别

```yaml
levels:
  error:   # 严重问题，Space 不合格
    - Space 没有根节点
    - Container 的 has_child 与实际不符

  warning: # 非严重问题，建议修复
    - title 为空
    - title 不符合预期格式

  info:    # 信息提示
    - 叶子节点数量统计
    - 最大深度统计
```

## 4. 模型产出

### 4.1 通用模型

```
┌─────────────────────────────────────────────────────────────┐
│                    Space 模型（跨 Space 通用）                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Space                                                     │
│  ├─ 必须有 Root 节点（depth=0, parent_token=""）           │
│  ├─ Root.has_child == false                                 │
│  ├─ Root.obj_type ∈ {docx, folder}                         │
│  │                                                          │
│  ├─ Container 节点（has_child=true）                        │
│  │  ├─ Container.title ≠ ""                                │
│  │  └─ Container.parent_token 指向存在的节点                │
│  │                                                          │
│  └─ Leaf 节点（has_child=false）                            │
│     ├─ Leaf.obj_type ∈ {docx, file}                        │
│     └─ Leaf.parent_token 指向存在的节点                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Space 类型特定规则（扩展）

```yaml
# 契约：Space 类型特定规则
space_types:
  议事档案:
    root_title_pattern: ".*首页"
    required_categories:
      - 议题
      - 议程
      - 决策
      - 计划
    document_pattern: "^\\d{4}年第\\d{1,2}周-.+"

  协作档案:
    root_title_pattern: "首页"
    container_patterns:
      - "任务管理"
      - "任务列表"
    document_pattern: "^\\d{4}年第\\d{1,2}周-任务\\d+-.+\\.docx$"

  图书馆:
    root_title_pattern: "编程语言"
    document_type: file
    document_extension: "pdf"

  软件库:
    root_is_file: true
    container_pattern: ".+"
    file_extension: "exe|pdf|docx"
```

### 4.3 验证规则 YAML

```yaml
# 通用验证规则（所有 Space 适用）
validation:
  rules:

    # 结构规则
    - name: 必须有根节点
      description: Space 必须有一个根节点（parent_token=""）
      assert: node_exists(depth=0)
      level: error

    - name: 根节点不能有子节点
      description: 根节点应该是叶子节点
      assert: node(depth=0).has_child == false
      level: error

    - name: Container 必须有子节点
      description: has_child=true 的节点必须有实际子节点
      assert: all(node.has_child == (child_count > 0) for node in containers)
      level: error

    - name: 父子关系完整性
      description: 除根节点外，所有节点的 parent_token 必须指向存在的节点
      assert: all_parent_tokens_valid()
      level: error

    # 命名规则
    - name: 节点标题不能为空
      description: 所有节点的 title 必须非空
      assert: all(node.title != "" for node in nodes)
      level: warning

    - name: Container 标题不能为空
      description: 分类节点必须有标题
      assert: all(node.title != "" for node in containers)
      level: error
```

### 4.4 契约文件设计

```yaml
# examples/feishu-wiki/contract.yaml

spec_version: 0.0.1
version: 1.0.0

# 资产定义
assets:
  feishu_wiki:
    title: 飞书知识库
    type: datasource
    category: feishu
    description: 飞书 Wiki 数字资产

# 验证配置
validation:
  # 通用规则（所有 Space 适用）
  common_rules:
    structural:
      - root_must_exist
      - root_must_be_leaf
      - container_has_children
      - parent_relation_valid

    naming:
      - title_not_empty
      - title_length_reasonable  # 1-200 字符

  # Space 类型特定规则（可选）
  space_specific_rules:
    enabled: true
    # 在 contract.yaml 同级目录创建 space/ 子目录
    # 每个 Space 一个 YAML 文件定义特定规则
```

## 5. 业务流程

### 5.1 验证执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                      验证执行流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 扫描发现                                                │
│     └→ 扫描 Space，获取 metadata.json                        │
│                                                             │
│  2. 加载契约                                                │
│     └→ 读取通用规则 + Space 特定规则（可选）                 │
```

## 6. 局限性说明

### 6.1 契约验证能做的

| 验证项 | 说明 |
|--------|------|
| ✅ 结构验证 | 节点存在性、父子关系完整性 |
| ✅ Container 验证 | has_child 准确性 |
| ✅ 命名验证 | title 非空、长度合理 |

### 6.2 契约验证不能做的

| 验证项 | 说明 | 解决方案 |
|--------|------|----------|
| ❌ 内容完整性 | 文档是否包含必要章节 | AI 验证层 |
| ❌ 命名格式 | 标题是否符合特定 Space 规范 | Space 特定规则 |
| ❌ 语义正确性 | 文档内容是否有逻辑错误 | AI 验证层 |
