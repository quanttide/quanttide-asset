# 议事档案资产目录规范

---

## 核心问题

议事档案的目录结构有什么规范？如何验证这些规范？

**目标**：通过资产目录结构，反推出组织的议事决策流程。

---

## 1. 议事档案结构

### 1.1 树形结构

```
议事档案
├── 议事档案首页              # Root，depth=0
├── 议题                      # Container，depth=1
│   ├── 议事规则              # Container，depth=2
│   │   ├── 2025年第49周-议事规则1-辩论
│   │   ├── 2025年第49周-议事规则2-决议
│   │   └── ...
│   ├── 议程                  # Container，depth=2
│   │   ├── 2025年第49周-议程2-新媒体运营
│   │   └── ...
│   ├── 决策                  # Container，depth=2
│   │   └── ...
│   └── 计划                  # Container，depth=2
│       └── ...
```

### 1.2 节点类型

| 类型 | depth | has_child | obj_type | 示例 |
|------|-------|-----------|----------|------|
| Root | 0 | false | docx | 议事档案首页 |
| Container | 1,2 | true | docx | 议题、议事规则、议程 |
| Leaf | 2+ | false | docx/file | 具体文档 |

### 1.3 元数据

```json
{
  "node_token": "xxx",
  "title": "2025年第49周-议事规则1-辩论",
  "obj_type": "docx",
  "parent_token": "父节点token",
  "has_child": false
}
```

## 2. 从资产反推业务流程

### 2.1 反推逻辑

```
分类层级 + 分类名称 + 叶子文档命名 = 业务流程
```

### 2.2 议事档案的反推结果

**观察到的分类结构**：

```
议题
├── 议事规则    → 制定讨论规则
├── 议程        → 安排讨论顺序
├── 决策        → 形成决策结论
└── 计划        → 制定执行计划
```

**反推出的议事流程**：

```
提出议题 → 制定规则 → 安排议程 → 做出决策 → 执行计划
    ↓           ↓          ↓         ↓         ↓
  议题      议事规则     议程      决策      计划
```

### 2.3 流程验证假设

如果 AI 能从资产结构反推出上述流程，且与实际议事流程基本一致，则说明建模有效。

## 3. 目录规范

### 3.1 结构规范

| 规范 | 说明 | 验证级别 |
|------|------|----------|
| 必须有根节点 | depth=0, parent_token="" | error |
| 根节点不能有子节点 | has_child=false | error |
| Container 必须有子节点 | has_child=true 时实际必须有子节点 | error |
| 父子关系完整 | parent_token 指向存在的节点 | error |
| 节点标题不能为空 | title ≠ "" | error |

### 3.2 分类规范

| 规范 | 说明 | 验证级别 |
|------|------|----------|
| 一级分类必须包含 | 议题/议程/决策/计划 | error |
| 分类层级不超过 3 层 | depth ≤ 3 | warning |

### 3.3 命名规范

| 规范 | 说明 | 验证级别 |
|------|------|----------|
| 叶子文档命名格式 | `YYYY年第N周-类型K-主题` | warning |
| 年份范围 | 2020-2030 | warning |
| 周数范围 | 1-52 | warning |

## 4. 契约设计

### 4.1 议事档案契约

```yaml
# contract.yaml

spec_version: 0.0.1
space_name: 议事档案

# 验证规则
validation:
  rules:

    # 结构规则
    - name: 必须有根节点
      assert: node_exists(depth=0)
      level: error

    - name: 根节点不能有子节点
      assert: node(depth=0).has_child == false
      level: error

    - name: 父子关系完整性
      assert: all_parent_tokens_valid()
      level: error

    - name: 节点标题不能为空
      assert: all(node.title != "" for node in nodes)
      level: error

    # 分类规则
    - name: 一级分类完整性
      description: 必须包含议题、议程、决策、计划
      assert: |
        has_category("议题") and
        has_category("议程") and
        has_category("决策") and
        has_category("计划")
      level: error

    - name: 分类层级不超过3层
      assert: max_depth <= 3
      level: warning

    # 命名规则
    - name: 叶子文档命名格式
      description: 应符合"YYYY年第N周-类型-主题"格式
      assert: matches(node.title, "^\\d{4}年第\\d{1,2}周-.+")
      where: node.has_child == false
      level: warning

    - name: 年份范围
      description: 年份应在 2020-2030 之间
      assert: extract_year(node.title) in range(2020, 2031)
      where: node.has_child == false
      level: warning

    - name: 周数范围
      description: 周数应在 1-52 之间
      assert: extract_week(node.title) in range(1, 53)
      where: node.has_child == false
      level: warning
```
