# 飞书 Wiki 资产目录规范

---

## 核心问题

飞书 Wiki 的目录结构有什么规范？如何验证这些规范？

**更高目标**：通过资产目录结构，反推出组织的业务流程。

---

## 1. 目录结构规范

### 1.1 树形结构

所有 Space 的目录都是一棵树：

```
Space（空间）
└── Root（根节点，depth=0）
    ├── Container（分类节点，has_child=true）
    │   ├── Container（子分类）
    │   │   └── Leaf（叶子文档）
    │   └── Leaf
    └── Leaf
```

### 1.2 节点类型

| 类型 | 特征 | obj_type | 说明 |
|------|------|----------|------|
| Root | parent_token="" | docx/folder | 根节点，Space 入口 |
| Container | has_child=true | docx/folder | 分类节点，用于组织结构 |
| Leaf | has_child=false | docx/file | 叶子节点，存储实际内容 |

### 1.3 元数据字段

```json
{
  "node_token": "节点唯一标识",
  "title": "节点标题",
  "obj_type": "docx | file | folder",
  "parent_token": "父节点 token（根节点为空）",
  "has_child": true | false
}
```

## 2. 从资产反推业务流程

### 2.1 核心思路

```
资产结构 ──分析──→ 业务语义 ──验证──→ 流程规范
```

通过分析 Space 的目录结构，可以推断出组织在做什么业务、采用什么流程。

### 2.2 反推示例

| Space | 目录结构 | 反推的业务流程 |
|--------|----------|----------------|
| 议事档案 | 议题 → 议事规则/议程/决策/计划 | **议事决策流程**：提出议题 → 制定规则 → 安排议程 → 做出决策 → 执行计划 |
| 协作档案 | 任务管理 → 任务列表 → 具体任务 | **任务协作流程**：管理任务 → 列出任务 → 执行任务 → 归档 |
| 图书馆 | 编程语言 → Android/C/Flutter/Java | **知识库分类**：按技术领域组织学习资料 |
| 软件库 | SSH软件/Python及IDE → 安装包 | **软件资产库**：按功能分类管理软件工具 |

### 2.3 反推逻辑

```
分类名称 + 分类层级 + 叶子文档内容 = 业务流程

示例：
- "议题" + "议事规则/议程/决策/计划" + 周报格式文档
- → 推断：这是一个"周期性的议事决策流程"

- "任务管理" + "任务列表" + "YYYY年第N周-任务K-主题.docx"
- → 推断：这是一个"周任务协作流程"
```

## 3. Space 类型与分类模式

### 3.1 固定分类模式

**Space**：议事档案

```
议事档案
├── 议题
│   ├── 议事规则
│   │   ├── 2025年第49周-议事规则1-辩论
│   │   └── 2025年第49周-议事规则2-决议
│   ├── 议程
│   │   └── 2025年第49周-议程2-新媒体运营
│   ├── 决策
│   └── 计划
```

**反推的业务流程**：
```
提出议题 → 制定规则 → 安排议程 → 做出决策 → 执行计划
    ↓           ↓          ↓         ↓         ↓
  议题       议事规则    议程     决策      计划
```

**规范**：
- 分类名固定：议题/议程/决策/计划/提案/审计/澄清/评估/谈判/报告/决议
- 叶子文档命名：`{年份}年第{周}周-{类型}{序号}-{主题}`

### 3.2 列表分类模式

**Space**：协作档案

```
协作档案
├── 首页
├── 任务管理
│   ├── 任务列表
│   │   ├── 2025年第9周-任务1-优化制冰机提示词.docx
│   │   └── 2025年第9周-任务2-优化制冰机数据报告.docx
│   └── 飞书任务管理教程
```

**反推的业务流程**：
```
任务管理 → 任务列表 → 具体执行 → 归档
    ↓          ↓          ↓         ↓
  管理视角   清单视角    执行文档   周报格式
```

**规范**：
- 分类名动态：如"任务管理"、"任务列表"
- 叶子文档命名：`{年份}年第{周}周-任务{序号}-{主题}.docx`

### 3.3 主题分类模式

**Space**：图书馆

```
图书馆
└── 编程语言
    ├── Android
    │   └── 第一行代码 Android.pdf
    ├── C Language
    │   └── ANSI C Standard.pdf
    └── Flutter
        └── Flutter实战.pdf
```

**反推的业务流程**：
```
按技术领域分类 → 管理学习资料 → 持续更新
```

**规范**：
- 按主题分类：编程语言/Android/C/Flutter/Java...
- 叶子节点是文件：书名.pdf

### 3.4 功能分类模式

**Space**：软件库

```
软件库
├── 开发者装机必备工具.docx
├── SSH软件
│   └── putty.exe
└── Python及IDE
    ├── pycharm-community-2022.1.3.exe
    └── python-3.10.11-amd64.exe
```

**反推的业务流程**：
```
按功能分类 → 管理软件工具 → 支持团队协作
```

**规范**：
- 功能分类：SSH软件/Python及IDE
- 叶子节点是安装包：.exe/.pdf

## 4. 目录规范总结

### 4.1 通用规范（所有 Space 必须遵守）

| 规范 | 说明 | 验证级别 |
|------|------|----------|
| 必须有根节点 | depth=0, parent_token="" | error |
| 根节点不能有子节点 | has_child=false | error |
| Container 必须有子节点 | has_child=true 时实际必须有子节点 | error |
| 父子关系完整 | parent_token 指向存在的节点 | error |
| 节点标题不能为空 | title ≠ "" | error |

### 4.2 流程验证规范（反推业务流程）

| 规范 | 说明 | 验证级别 |
|------|------|----------|
| 分类层级符合预期 | 深层级太多说明分类过细 | warning |
| 叶子文档命名一致 | 同类文档命名格式统一 | warning |
| 周格式文档按周归档 | `YYYY年第N周` 格式应归档到对应周 | warning |

### 4.3 类型特定规范

| Space 类型 | 分类模式 | 叶子文档命名 | 验证级别 |
|------------|----------|--------------|----------|
| 议事档案 | 固定分类 | `YYYY年第N周-类型-主题` | warning |
| 协作档案 | 列表分类 | `YYYY年第N周-任务K-主题.docx` | warning |
| 图书馆 | 主题分类 | `书名.pdf` | warning |
| 软件库 | 功能分类 | `软件名.exe` | warning |

## 5. 契约设计

### 5.1 通用契约（所有 Space 适用）

```yaml
# contract.yaml

spec_version: 0.0.1

# 通用验证规则
validation:
  rules:

    # 结构规则
    - name: 必须有根节点
      assert: node_exists(depth=0)
      level: error

    - name: 根节点不能有子节点
      assert: node(depth=0).has_child == false
      level: error

    - name: Container 必须有子节点
      description: has_child=true 的节点必须有实际子节点
      assert: all(node.has_child == (child_count > 0) for node in containers)
      level: error

    - name: 父子关系完整性
      assert: all_parent_tokens_valid()
      level: error

    # 命名规则
    - name: 节点标题不能为空
      assert: all(node.title != "" for node in nodes)
      level: error
```

### 5.2 流程验证契约（反推业务流程）

```yaml
# contract_flow.yaml

spec_version: 0.0.1

# 流程验证规则：从资产结构反推业务流程

validation:
  rules:

    # 议事档案流程验证
    - name: 议事档案-流程完整性
      description: 议事档案应包含完整的议事流程节点
      assert: |
        has_category("议题") and
        has_category("议程") and
        has_category("决策") and
        has_category("计划")
      level: warning

    - name: 议事档案-叶子文档格式
      description: 叶子文档应符合"YYYY年第N周-类型-主题"格式
      assert: matches(node.title, "^\\d{4}年第\\d{1,2}周-.+")
      where: space_name == "议事档案" and node.has_child == false
      level: warning

    # 协作档案流程验证
    - name: 协作档案-任务格式
      description: 任务文档应符合"YYYY年第N周-任务K-主题.docx"格式
      assert: matches(node.title, "^\\d{4}年第\\d{1,2}周-任务\\d+-.+\\.docx$")
      where: space_name == "协作档案" and node.obj_type == "file"
      level: warning
```
